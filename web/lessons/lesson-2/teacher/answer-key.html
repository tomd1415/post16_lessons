Answer guidance

a01 — Formal vs natural language

  • Quick checks: formal language = Python; computers prefer precision; “move forward a bit” is ambiguous.
  • Step 2 table: formal = fixed rules; examples: Python, maths notation; natural example: chat/story.
  • Task A radios: 1=F, 2=N, 3=F, 4=N.
  • Task B numbered inputs: look for measurable steps (quantities/timings).
  • Task C: if ambiguity flagged = Yes, ensure the rewritten line is clearer.
  • Misconception: formal = polite; use “Can a CPU guess?” prompt.

a02 — Source vs machine code

  • Quick checks: humans read source; machine code is binary; compiler/assembler translates.
  • Step 2 table: source example e.g., print('hi'); machine example = binary bytes.
  • Task A radios: 1=M, 2=S, 3=S (assembly), 4=S/comment.
  • Pipeline quiz: correct = compile/assemble; reason = translates to CPU-ready bits.
  • Snippet line: “CPU loads 2 and 3, adds, stores result”; optional opcode noted.
  • Misconceptions: “CPU runs .py directly”; “comments execute”.

a03 — Software development steps

  • Quick checks: Analysis first; testing during/after programming; enhancement after release.
  • Step 2 table: concise phase definitions.
  • Ordering dropdowns: Analysis → Design → Programming → Testing → Enhancement.
  • Artefact matches: Analysis→requirements; Design→flowchart/wireframe; Programming→working code; Testing→test report; Enhancement→change log/backlog.
  • Mini-case lines: expect one short action per phase; risks: logic bugs, missed requirements, perf issues.

a04 — Specification for a game

  • Quick checks: specs must be clear/testable; functional = what it does; non-functional = qualities (speed, accessibility).
  • Step 2 inputs: functional example = gameplay behaviour; non-functional = performance/accessibility.
  • Spec table: goal (win condition), inputs (controls), outputs (score/sound), rules (collisions/lose life), constraints (load time, keyboard-only).
  • Acceptance checks: “Score resets to 0 on new game”, “Player loses 1 life on collision”, “Level loads < 2s”.
  • Watch for vague words: fun, cool — replace with measurable behaviour.

Evidence to look for

  • Completed quick checks (auto-marked).
  • Structured notes for specification with at least three acceptance statements.
  • Reflections noting translation risks (compiler errors, ambiguous requirements).
